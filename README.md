# RAG Application

An AI-powered document Q&A, web search, and data analysis application. Upload documents or data files, connect external databases, and ask questions in natural language. A LangGraph agent dynamically routes each query to your uploaded documents, live web search, DuckDB-powered data analysis, or the LLM's own knowledge ‚Äî then synthesises a grounded answer with source attribution.

## Features

- **Multi-format document pipeline** ‚Äî upload PDF, DOCX, TXT, MD, JSON, XLSX, CSV. Files are chunked, embedded (768-dimensional Gemini vectors), and stored in PostgreSQL with pgvector.
- **Agentic RAG** ‚Äî a LangGraph agent classifies each query and routes it to: document retrieval, live web search (Tavily), both sources combined, data analysis (DuckDB), or plain LLM knowledge. Each response carries a source badge.
- **Streaming responses** ‚Äî assistant answers stream token-by-token via Server-Sent Events; status updates show which agent node is running in real time.
- **Multi-turn chat** ‚Äî full conversation history, multiple named threads per user, pin threads, edit-and-regenerate, copy. Thread titles auto-generated by the LLM.
- **Voice input** ‚Äî microphone button in the chat input uses the browser SpeechRecognition API to transcribe speech.
- **Dark / light theme** ‚Äî toggled from the Profile page; preference persisted in localStorage with no flash on load.
- **User accounts** ‚Äî signup with email verification, JWT auth (httpOnly cookies, refresh token rotation), profile with avatar, password change, token usage stats, account deletion.
- **Data analysis** ‚Äî upload CSV, Excel, Parquet, TSV, or JSON files (max 20 MB), or connect PostgreSQL, MySQL, SQLite, Amazon S3, GCS, Azure Blob, or a REST API. Ask questions in natural language; the agent generates DuckDB SQL, executes it, and renders results as an interactive paginated table (table first, then a 2-sentence LLM insight) with SQL transparency and CSV export. Connection credentials are Fernet-encrypted at rest and never exposed in API responses.
- **Redis caching** ‚Äî DuckDB query results cached for 600 s; document counts cached for 60 s to skip Postgres on every routing decision. Cache is invalidated automatically on file delete and document changes. Graceful fallback if Redis is unavailable.
- **Nginx proxy cache + CDN headers** ‚Äî `/_next/static/` assets cached at the proxy for 365 days with `immutable` headers; API and SSE endpoints explicitly bypass caching. Security headers (`X-Content-Type-Options`, `X-Frame-Options`, `X-XSS-Protection`, `Referrer-Policy`) on all responses.
- **Production Docker setup** ‚Äî nginx reverse proxy, Redis, multi-stage Docker builds, automatic database migrations on startup, health checks on all services.

## Quick Start

**1. Clone the repository:**

```bash
git clone <repo-url>
cd rag-application
```

**2. Configure environment variables:**

```bash
cp .env.example .env
```

Open `.env` and fill in the required API keys (see [Environment Variables](#environment-variables) below).

**3. Start all services:**

```bash
docker compose up -d
```

This builds images, runs database migrations automatically, and starts postgres, redis, backend, frontend, and nginx. First run takes 3‚Äì5 minutes while Docker builds the images.

**4. Open the app:**

```
http://localhost
```

Create an account and start chatting. With `REQUIRE_EMAIL_VERIFICATION=false` (the default in the provided `.env.example`) you can sign in immediately without clicking a verification link.

**5. Verify everything is healthy:**

```bash
docker compose ps
# All five services should show (healthy)

curl http://localhost/health
# ‚Üí {"status":"healthy","database":"connected","version":"1.0.0"}
```

### Data flow for a chat message (streaming)

1. User sends a message via the frontend.
2. `POST /api/chat-threads/{id}/messages/stream` opens an SSE connection.
3. `router_node` checks the document count (Redis cache, TTL 60 s) and asks Gemini to classify the query ‚Üí `status` event sent.
4. Based on route: `retrieval_node` searches pgvector, `websearch_node` calls Tavily, or `data_analysis_node` generates and executes DuckDB SQL (results cached in Redis, TTL 600 s) ‚Üí `status` events sent.
5. `synthesis_node` builds a context block (including actual data rows for data analysis queries) and streams Gemini output token-by-token ‚Üí `token` events sent.
6. The assistant message (with token counts, source label, and optional `data_analysis_result`) is saved to the database ‚Üí `done` event sent.

### Data analysis response layout

For data analysis queries, the frontend renders in this order:
1. **SummaryStats** ‚Äî source badges and row count
2. **SqlDisplay** ‚Äî collapsible SQL block with line numbers and copy button
3. **DataTable** ‚Äî paginated table (25 rows/page) with numeric formatting and CSV download
4. **LLM insight** ‚Äî 2-sentence synthesis (most important finding + pattern/implication)

## Development Guide

### Running the backend locally (without Docker)

Useful when iterating quickly on backend code with a local Postgres instance.

```bash
cd backend

# Create venv (once)
python -m venv venv

# Activate
# Windows
venv\Scripts\activate

source venv/bin/activate

# Install dependencies
pip install -r requirements.txt

# Set in .env:
#   POSTGRES_HOST=localhost
#   REDIS_URL=redis://localhost:6379   (requires Redis running locally)
#   NEXT_PUBLIC_API_URL=http://localhost:8000

# Start Redis (if not already running)

# Run migrations
alembic upgrade head

# Start the dev server (with hot reload)
uvicorn app.main:app --reload
# ‚Üí http://localhost:8000
```

Always activate the venv before running any Python commands in `backend/`.

Set `NEXT_PUBLIC_API_URL=http://localhost:8000` in `.env` so the browser can reach the backend directly.

## Database Migrations

All schema changes go through Alembic. The backend `entrypoint.sh` runs `alembic upgrade head` automatically on every container start, so migrations are never missed.

```bash
# Apply all pending migrations (also runs automatically on startup)
docker exec rag-application-backend-1 alembic upgrade head

# Or locally with venv activated
cd backend && alembic upgrade head

# Roll back one step
alembic downgrade -1

# Check current revision
alembic current

# Show full history
alembic history

# Generate a new migration after changing SQLAlchemy models
alembic revision --autogenerate -m "describe the change"
```

**Never** call `Base.metadata.create_all()` or modify the schema manually ‚Äî Alembic is the only mechanism.

4. `d8e9f0a1b2c3` ‚Äî profile picture storage (binary + content type) on users
5. `e1f2a3b4c5d6` ‚Äî refresh_tokens table with SHA-256 token hashes

## Deployment Guide

The application is container-ready and can be deployed to any Linux server with Docker.

### Production `.env` changes

```bash
POSTGRES_HOST=postgres            # keep as-is (Docker service name)
ENVIRONMENT=production            # enables secure cookies, disables debug links
REQUIRE_EMAIL_VERIFICATION=true   # enforce email verification
EMAIL_FROM=noreply@yourdomain.com # must be a Resend-verified domain
LOG_LEVEL=warning                 # reduce log volume
```

### Resend domain verification

In production, `EMAIL_FROM` must use a domain you own and have verified in the Resend dashboard. Resend's `onboarding@resend.dev` sender only works in test mode to a single recipient.

### SSL with Certbot (optional)

To add HTTPS, place Certbot-issued certificates on the host and update `nginx/nginx.conf` to listen on port 443. The nginx config is mounted as a volume and can be updated without rebuilding the image.

## Troubleshooting

**Port 80 already in use:**
```bash
docker compose ps           # check what's using port 80
# or on Linux: sudo lsof -i :80
```
Stop the conflicting service and re-run `docker compose up -d`.

**"unhealthy" container on startup:**
```bash
docker compose logs backend   # check for startup errors (migrations, config)
docker compose logs frontend
docker compose logs redis
```
The most common cause is a missing or incorrect environment variable. Verify `.env` has all required keys.

**Database connection errors:**
`POSTGRES_HOST` must be `postgres` when running inside Docker Compose. If running the backend locally outside Docker, set it to `localhost`.

**Redis connection errors:**
The backend falls back gracefully to cache-miss behaviour if Redis is unavailable ‚Äî the app still works, just without caching. If you want caching during local dev, run `docker compose up -d redis` and set `REDIS_URL=redis://localhost:6379` in `.env`.

**Email not sending:**
1. Verify `RESEND_API_KEY` is correct in `.env`.
2. Check the backend logs for Resend error messages.
3. Set `REQUIRE_EMAIL_VERIFICATION=false` to bypass email verification during local development.
4. In production, ensure `EMAIL_FROM` uses a Resend-verified domain.

**Document processing stuck in "processing" status:**
Processing runs as a FastAPI `BackgroundTask`. If the backend restarts mid-processing, the document remains stuck. Delete the document via the UI and re-upload to trigger reprocessing.

**Data file analysis fails or returns stale results:**
DuckDB query results are cached in Redis for 600 s. If you re-upload a file with the same name, delete the old file first ‚Äî deleting a file automatically invalidates its cache entries.

**Rebuilding clears the database:**
The `docker compose down -v` flag removes volumes including the Postgres and Redis data volumes. Use `docker compose down` (without `-v`) to stop services while preserving data.
